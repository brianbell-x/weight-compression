Does it scale?

Not a 30B fluke: the same code holds from 30B to 550B.

The win comes from how the Nemotron-3 family stores BF16, not from any one model's size. Run the identical codec across the family with a bounded-disk validator (one shard in memory at a time, no full download) and the reduction stays flat within about a point across an 18× span.

How far can lossless go?

Within half a bit of the information-theoretic limit.

There is a hard floor for lossless BF16: you cannot store a weight in fewer bits than its Shannon information. Our random-access form sits right on top of that floor. Going lower means discarding bits, which changes the outputs.

The measured entropy floor for these weights is ~10.8 bits/weight. Our fixed-width, matmul-readable code reaches 11.30. The ~0.5-bit gap is the price of random access. INT8's 8.0 lives below the lossless floor only because it throws information away.

How it works

Split each weight where the information actually is.

A BF16 number is 16 bits: 1 sign, 8 exponent, 7 mantissa. We measured how much real information each part carries across the model. The answer is lopsided.

Why a tiny codebook works

The sign+exponent field is hyper-concentrated: 16 entries cover 99.9% of all weights. The rare leftovers ride a sparse escape stream (~0.1% of weights at this layout), so nothing is lost.

Decode is a table lookup, in-register

# per weight, at matmul time, no VRAM re-inflation
sign_exp = codebook[idx]            # 4-bit index → 9-bit field
bf16     = (sign_exp << 7) | mantissa   # mantissa kept raw

# rare escapes patched from a sparse side stream
if is_escape: bf16 = escapes.next()

Every weight has a known bit-offset, so thousands of GPU lanes fetch weight (i, j) in parallel. The full BF16 exists only in a register, never back in VRAM.

How we tested it

Proof is an exact round-trip, not a quality score.

Lossless means the reconstructed bytes equal the originals. We check that with SHA-256 and bit-equality, not perplexity: there is no quality metric left to move.

A fused kernel reads the narrow form on-device and computes the matmul directly, BF16 rebuilt only in registers. Output matches full BF16 to float-rounding, so the compressed weights run without ever being re-inflated in memory.

exact on-device

KL(compressed ‖ original) = 0 exactly. Identical inputs + bit-identical weights + identical code ⇒ bit-identical logits, deterministically.

Why it's different

Lossless and fusible, not one or the other.

vs. quantization (INT8 / INT4)

  • Same: a fixed-width code the matmul reads directly, rebuilt in-register.
  • Different: we change zero bits of output. Quantization must measure that quality survived; here quality is preserved by definition, so no eval exists that can find a difference.

vs. variable-length lossless

  • Same: exact reconstruction, ~30% smaller on disk.
  • Different: entropy-coded schemes have no fixed bit-offset, so they must inflate back to full-width BF16 in VRAM before the matmul. Ours is fixed-width → random-access → the math reads it in place.
