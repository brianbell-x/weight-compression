Q-01 · Isn’t this just zip? Or DFloat11?

Zip-style tools (zstd) get ~22% — then everything must be unpacked before the GPU can touch a weight. Published lossless codecs (DFloat11, ZipNN) reach ~30–33% with variable-length codes — no weight has a fixed address, so every read pays a decode toll. Here every weight keeps a fixed address: the GPU does its math on the compressed bytes directly.

Q-02 · Why not just quantize?

Quantization deletes bits. Usually fine — never provable: you benchmark and hope. This keeps every bit, so the outputs are identical by construction. If 8-bit quantization (Q8) satisfies you, use it; this is for when “probably fine” isn’t enough. (It won’t stack on quantized files — quantizing spends the exponent redundancy this feeds on.)

Q-03 · What’s the catch?

The speed win is bandwidth-bound: the kernel — the small GPU program that reads weights and multiplies — takes 24% less time on a 4090, measured on the −25% sibling form. On an H100 it ties: the in-register unpacking tops out near the speed plain BF16 already reads at, so you keep the full memory saving and gain no time. And this is research, not a product: no llama.cpp or vLLM loader exists yet.

PLATE 1 ANATOMY OF A WEIGHT · BF16 BIT-FIELD SHEET 2 / 5

Half of every weight is coin flips. The other half is mostly air.

A BF16 weight is 16 bits of scientific notation: a sign, an exponent (the number’s size), a mantissa (its fine detail). Measured across all 31.6 billion weights, the mantissa is a hair from pure noise — while the exponent field is so repetitive it is mostly empty space. Essentially every point of compression comes from that emptiness.

02 THE SPLIT

Split the repetitive half from the noise. Compress it alone.

Split each tensor into an exponent plane and a mantissa plane; entropy-code the first — fewer bits for more frequent values — and ship the second verbatim. 32%, bit-exact, squarely on the published plateau (DFloat11 ~30%, ZipNN ~33%). And here is the wall they all share: a variable-length stream gives no weight a fixed address, so the math cannot read it in place — it must be re-inflated first, wholesale or block-by-block, paying in time. A great archive. Useless at runtime.

03 THE BRIDGE

Trade 2 points of storage for random access.

Re-express the same structure as a fixed-width 4-bit code per weight — an index into that tensor’s codebook — plus a small escape list for the values that miss it. Now every weight has a computable address: the GPU reads the compressed bytes in place and rebuilds BF16 inside a register. Fewer bytes means faster in a bandwidth-bound job: a fused kernel over the byte-split sibling (12 bits/weight, receipts row 03) took 24% less time than plain BF16 on an RTX 4090; the −30% headline form shares the same fixed-address layout but hasn’t been re-benchmarked. On an H100 the kernel ties BF16 — full VRAM cut, zero speed tax — where DFloat11’s variable-length decode runs 1.4–2x slower at small batch (its own numbers).

Find the wall on purpose.

PLATE 4 OPEN ITEMS · REPRODUCTION · COLOPHON SHEET 5 / 5

What’s still open.

30 is where the easy structure ran out, not where the project ends.

Open item 01 · Layout OPEN

The gap we can see

A storage-only codec reaches 33.7%; random access costs points back. The gap is closing: a verified container with per-weight random access sits at 31.89%, and a tile format — coding small blocks of weights instead of single weights — measures 33.2% end-to-end, 33.6% with a sharper coder: essentially the ~34% ceiling, realized and round-trip-verified, decoding one block at a time. What’s open is carrying that block-addressable form into a kernel. Step 3 already solved one problem of exactly this species.

Status: layout problem · Per-weight best: 31.89% · Block-granular best: 33.6% · Precedent: step 3
Open item 02 · Hypothesis OPEN

The wall that already cracked — a hairline

“The mantissa is random” only ever meant “random to every tool we aimed at it.” Proof: a fresh look found a faint, exponent-driven bias in its top bit — measured, then cashed for ~0.04 bits/weight. A hair off the total; the ~34% limit stands. But the lesson does too: “random” is a verdict on our tools, never on the bits. The next hypothesis is welcome.

Status: bias found & cashed (+0.04 bits/weight) · Stakes: the next crack
