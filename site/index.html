Lossless BF16
Compression Research

A full GLM-5.2 scan found 30.168% K15 charged-format accounting. A separate byte-split representation was decoded bit-for-bit across all 59,509 BF16 tensors at 24.967% reduction. Those are distinct evidence classes.

How it works

[ FIG. 01 ]
01 / Where the K15 estimate comes from

Most weights share a few sign-exponent symbols

Every BF16 weight is 16 bits: a sign, 8 exponent bits, 7 mantissa bits. In a trained model the exponents are wildly repetitive; a handful of values cover nearly every weight.

The K15 accounting replaces the 9-bit sign-and-exponent symbol with a 4-bit code into a 15-entry table. Rare 9-bit symbols go to an exact escape stream; the 7-bit mantissa stays raw. With all side costs charged, the full scan prices this layout at 11.173 bits per weight.

02 / What was decoded exactly

The byte-split path round-trips

The streamed validator reconstructed the high byte from its codebook, indices, and escape stream, kept the low byte verbatim, and compared every BF16 tensor with the source.

All 59,509 tensors matched bit-for-bit. That decoded representation costs 12.005 bits per weight, or 24.967% less than BF16. The 30.168% K15 layout is separately charged accounting and was not independently decoded at GLM scale.

03 / Runtime evidence and open work

A historical dense-only microbenchmark

One A40 test reconstructed BF16 weights from packed codes in registers for a single [2048×6144] GLM tensor tiled K=476. It measured 14.31 ms versus 19.52 ms BF16, or 0.733 times the BF16 GEMV time. Correction was excluded; later H100 tests showed no speedup. This is neither full-model throughput nor an exact-path result.

An exact lossless speedup, a physical GLM-scale K15 container, and end-to-end inference integration therefore remain open. See the full measurement contract and pending evidence fields.

04 / Prior work and distinction

The exponent insight and fused concept are prior art

ZipNN and DFloat11 established lossless BF16 exponent compression in this size range. DFloat11 uses variable-length Huffman codes and restores BF16 weights before matrix multiplication. ZipServ is the closest prior runtime design: it already demonstrated fixed-length coding with direct register reconstruction. This experiment uses a different representation, per-tensor 4-bit codes over 15 joint sign-and-exponent symbols with sparse exact escapes, and does not claim the shared exponent redundancy, roughly 11-bit range, or fused concept as new.

[ FAQ ] Practical boundaries

Questions this result should answer

The short version: the codec is exact for the representation it receives, but combination, transfer, and runtime claims remain qualified by what has actually been tested.

“Possible” is not “demonstrated.” Each answer separates the mechanism from the current evidence.

01 / CombinationCan this be combined with quantization?

In principle, yes—but that would be a different experiment. If a model is quantized first, an exact codec can compress the resulting quantized representation, but “lossless” then means bit-exact relative to those quantized weights, not to the original BF16 checkpoint. The information discarded by quantization cannot be restored.

This method is designed around structure in BF16 sign, exponent, and mantissa fields. INT8, INT4, and custom quantized layouts have different symbol distributions, packing, metadata, and kernel contracts, so the same compression rate or runtime behavior should not be assumed. A combined runtime path would also need to charge both formats’ metadata and include dequantization in every timing.

Evidence status · Quantization is outside this project’s current scope and the combination has not been measured.

02 / Prior workWhat is the difference between this and DFloat11?

Both are exact BF16 compression methods built around low-entropy exponent information, but the representations and runtime contracts differ. DFloat11 keeps sign and mantissa bits uncompressed, Huffman-codes the exponent with variable-length codes, and decompresses the required weights back to BF16 before matrix multiplication. Its published system batches decompression at transformer-block level and discards the temporary BF16 matrices afterward.

Track 1 here uses fixed-width byte-split planes plus exact sparse BF16 correction. Track 2 targets a per-tensor 4-bit code over 15 joint sign-and-exponent symbols, keeps the 7-bit mantissa, and uses exact escapes. The runtime goal is random access from the compressed resident form, reconstructing BF16 values only transiently rather than materializing a full-width matrix.

Evidence status · DFloat11 has published multi-model GPU results. This project’s full-GLM exact runtime and K15 physical container remain pending; no superiority claim is made.

03 / TransferCan this be applied to other models? Is it architecture-specific?

The codec is tied to the weight representation more than to one transformer architecture. It can be tested on other BF16 models without retraining when their tensors contain enough reusable sign/exponent structure. Compression rate, escape rate, and the best per-tensor codebook can still vary by model, layer, and checkpoint.

The runtime integration is more architecture-aware: module selection, tensor shapes, sharding, expert routing, and GPU topology all affect coverage, memory, and speed. That is why Gemma 4 31B is the development target and full GLM-5.2 is the final proof target. Any additional model or architecture receives a new source/artifact identity and must pass its own exact inverse, GPU correctness, residency, accounting, and runtime gates.

Claim boundary · No compression percentage, VRAM saving, or throughput result transfers to an untested model by assumption.
